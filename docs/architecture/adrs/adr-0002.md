# ADR-0002: Storage Independence via Adapter Pattern

**Status:** Accepted
**Date:** 2026-07-21
**Deciders:** Core maintainers

## Context

The system must persist canonical and derived data across multiple storage engines: relational, graph, search, vector, object, and cache. The system must support replacing any storage engine without changing the domain model.

## Decision

Each storage engine is accessed through an adapter. Adapters implement a common interface. The domain model never depends on a specific storage technology. Replacing a storage engine requires implementing a new adapter, migrating data (if canonical), and rebuilding derived data. The domain model, the pipeline, and all business logic remain unchanged.

## Alternatives Considered

### Option 1: Single relational database
- Pros: Simple, familiar, ACID transactions
- Cons: One database cannot optimize for all access patterns. Graph traversal, vector similarity, and full-text search are poorly served by relational queries.
- Why not chosen: Violates polyglot persistence. Forces all access patterns through a single engine.

### Option 2: Direct storage integration
- Pros: Simpler code, fewer abstractions
- Cons: Couples the domain model to specific storage engines. Replacing a storage engine requires rewriting business logic.
- Why not chosen: Violates storage independence.

### Option 3: Adapter pattern (chosen)
- Pros: Storage engines are interchangeable. The domain model is independent. New storage engines are added as plugins.
- Cons: Adds an abstraction layer. Requires maintaining adapter implementations.
- Why chosen: Enables storage independence while supporting polyglot persistence.

## Consequences

### Positive
- Any storage engine may be replaced without changing the domain model.
- New storage engines are added as plugins without modifying the core.
- The domain model remains pure and technology-independent.

### Negative
- The adapter abstraction adds complexity.
- Each adapter must be maintained and tested.
- Cross-engine consistency requires event-driven coordination.

### Risks
- Adapter interfaces may be too generic or too specific. Mitigation: interfaces are refined through real usage and versioned.
