# ADR-0003: Entity Component Model for Knowledge Entities

**Status:** Accepted
**Date:** 2026-07-21
**Deciders:** Core maintainers

## Context

The system must represent diverse knowledge entities: people, organizations, papers, tools, concepts, decisions, and more. Traditional object-oriented design uses inheritance hierarchies, which create rigid, fragile, and duplicative class structures.

## Decision

Entities are assembled from composable components rather than defined through inheritance hierarchies. Each entity is a container for components. Each component represents one aspect of the entity. Entities acquire behavior through component assembly, not class inheritance.

This follows the Entity Component System (ECS) pattern from game engine architecture, adapted to knowledge management.

## Alternatives Considered

### Option 1: Inheritance hierarchy
- Pros: Familiar OOP pattern, type safety through class hierarchies
- Cons: Rigid hierarchies, code duplication, fragile base classes, diamond problem. Adding a new entity type requires creating new classes and modifying existing ones.
- Why not chosen: Creates rigid, fragile, and duplicative code structures.

### Option 2: Type tags with shared behavior
- Pros: Simpler than full ECS, flexible
- Cons: Behavior is scattered across type-specific code paths. Adding new behavior requires modifying existing code.
- Why not chosen: Introduces implementation leakage.

### Option 3: Entity Component Model (chosen)
- Pros: Flexible, composable, extensible. Adding new entity types requires only adding a new type configuration. Adding new behavior requires only adding a new system.
- Cons: More complex than inheritance. Components are data-only; behavior lives in systems.
- Why chosen: Eliminates inheritance problems while enabling unlimited extensibility.

## Consequences

### Positive
- Adding new entity types requires no code changes, only configuration.
- Adding new component types requires no changes to existing entities.
- Systems operate on component sets, not entity types, enabling flexible behavior.

### Negative
- The component model is less familiar than inheritance.
- Component validation must be performed at the application level.
- Querying entities by component set is more complex than querying by type.

### Risks
- Component sprawl (too many component types) may make the model hard to understand. Mitigation: core component types are well-defined; custom types are added sparingly.
