# ADR-0004: Event-Driven Derivation Pipeline

**Status:** Accepted
**Date:** 2026-07-21
**Deciders:** Core maintainers

## Context

The system must propagate canonical data changes to derived artifacts: search indexes, embeddings, graph projections, and caches. The propagation must be asynchronous, independent, and resilient.

## Decision

Every meaningful modification to canonical data emits an event. Events are processed asynchronously through the derivation pipeline. Each derived artifact is updated by an independent event handler. Handlers are idempotent and isolated.

## Alternatives Considered

### Option 1: Synchronous updates
- Pros: Strong consistency. Derived data is always up-to-date.
- Cons: Slow. Every canonical change blocks until all derived data is updated. A failure in one derivation blocks all others.
- Why not chosen: Blocks the pipeline. Prevents parallelism. Reduces throughput.

### Option 2: Periodic batch updates
- Pros: Simple. Updates derived data on a schedule.
- Cons: Derived data is stale between batches. No real-time consistency. Batch failures are hard to diagnose.
- Why not chosen: Introduces unacceptable staleness. Prevents real-time views.

### Option 3: Event-driven derivation (chosen)
- Pros: Asynchronous, parallel, resilient. Each handler is independent. Failures in one handler do not affect others. Derived data is updated near-real-time.
- Cons: Derived data is eventually consistent, not instantly consistent. Adds event infrastructure complexity.
- Why chosen: Balances consistency, performance, and resilience.

## Consequences

### Positive
- Each derivation type scales independently.
- Handler failures do not affect other handlers or canonical data.
- The pipeline is extensible: new derivation types are added as new event handlers.
- Events support replay for rebuilding derived data after failures.

### Negative
- Derived data is eventually consistent. Views may briefly show stale data.
- Event ordering across entities is not guaranteed.
- The event infrastructure adds complexity.

### Risks
- Event loss could cause derived data to drift from canonical data. Mitigation: events are durable and replayable. Dead letter queues capture unprocessable events.
