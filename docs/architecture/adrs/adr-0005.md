# ADR-0005: Compiler-Inspired Architecture

**Status:** Accepted
**Date:** 2026-07-21
**Deciders:** Core maintainers

## Context

The system must process heterogeneous knowledge through a sequence of transformations: from raw external data to canonical representation to derived artifacts to rendered views. The transformation pipeline must be deterministic, composable, and testable.

## Decision

The system is modeled as a deterministic knowledge compiler. Information enters as source material, passes through import, parsing, normalization, and canonical representation, and is then compiled into derived artifacts optimized for specific access patterns.

The compiler analogy provides a well-understood mental model for the pipeline:

| Compiler Stage | Knowledge OS Stage |
|---------------|-------------------|
| Lexical analysis | Import layer (format detection, extraction) |
| Parsing | Parsing layer (structural analysis) |
| Semantic analysis | Normalization layer (entity resolution, identity) |
| Intermediate representation | Canonical entity model |
| Optimization | Derivation layer (index generation, embedding) |
| Code generation | Presentation layer (view rendering) |
| Executable | Rendered knowledge projection |

## Alternatives Considered

### Option 1: Request-response pipeline
- Pros: Simple, familiar web architecture pattern.
- Cons: Tight coupling between stages. No natural support for derivation or projection. No clear separation between canonical and derived data.
- Why not chosen: Does not enforce the canonical/derived distinction.

### Option 2: Actor model
- Pros: Each entity is an independent actor. Natural parallelism.
- Cons: Actors manage their own state, which may conflict with the canonical model. No clear pipeline structure.
- Why not chosen: Does not provide the deterministic transformation pipeline the system requires.

### Option 3: Compiler-inspired architecture (chosen)
- Pros: Deterministic, composable, testable. Each stage has one responsibility. Stages communicate through explicit intermediate representations. The canonical model is the intermediate representation.
- Cons: Adds pipeline infrastructure. Each stage must be implemented and tested.
- Why chosen: Provides a proven architectural pattern for sequential transformations with well-defined intermediate states.

## Consequences

### Positive
- The pipeline is deterministic: same input produces same output.
- Each stage is independently replaceable and testable.
- The canonical model serves as the intermediate representation, enabling incremental compilation.
- Derived artifacts are disposable and rebuildable, like compiled object files.

### Negative
- The pipeline adds latency for multi-stage processing.
- Each stage must be implemented and maintained.
- The compiler analogy may not be immediately obvious to all contributors.

### Risks
- Pipeline stages may become too tightly coupled. Mitigation: stages communicate through explicit contracts, not shared state.
