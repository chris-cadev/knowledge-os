# ADR-0006: Entity Resolution as the Critical Quality Layer

**Status:** Accepted
**Date:** 2026-07-22
**Deciders:** Core maintainers

## Context

The system must transform raw input (documents, imports, API data) into canonical entities. The quality of the knowledge graph depends not on the model itself, but on the accuracy of entity resolution — ensuring that "OpenAI," "OpenAI Inc.," and "the company" resolve to one canonical entity.

2026 enterprise evidence confirms this is the critical quality layer:

- Children's Medical Center Dallas found 22% of patient records were duplicates before proper entity resolution (Learning from Data, June 2026).
- A paper from NTU, Nanjing University, and Mila found that LLMs consistently produce duplicate entities in GraphRAG pipelines. Reducing graph size by 40% through entity resolution improved QA performance across all variants.
- "The extraction step works. What doesn't work automatically is everything that afterward" (Upside, June 2026).
- "Extraction is the part that demos, resolution is the part that matters" (Dreaming Press, June 2026).

Without robust entity resolution, the knowledge graph accumulates compounding noise. Duplicate entities corrupt path analysis, degrade search results, and silently undermine AI outputs.

## Decision

The Normalization Layer (Layer 3) is the critical quality layer. Entity identification, duplicate detection, canonical ID assignment, and metadata normalization receive first-class engineering attention. The pipeline spine is: **Import → Extract → Resolve → Store → Connect → Search**.

Entity resolution is continuous, not one-time. New imports are matched against existing entities incrementally. The system never assumes that a new import represents a new entity without checking for existing matches.

## Alternatives Considered

### Option 1: Resolution as post-processing
- Pros: Simple. Import first, resolve later.
- Cons: Duplicates accumulate before resolution. Downstream layers (search, relationships) operate on noisy data. Resolution failures compound.
- Why not chosen: Quality degrades before correction is applied.

### Option 2: Resolution at query time
- Pros: No upfront cost. Query handles deduplication.
- Cons: Every query pays the resolution cost. Inconsistent results depending on query path. Complexity moves to the hardest layer.
- Why not chosen: Unacceptable latency and inconsistency.

### Option 3: Resolution at import time (chosen)
- Pros: Canonical data is clean from the start. Downstream layers operate on resolved entities. Quality is maintained by construction.
- Cons: Import is slower. Resolution requires matching against existing data. False positives may merge distinct entities.
- Why chosen: Quality is maintained by construction, not corrected after the fact.

## Consequences

### Positive
- The canonical model is clean from the first import.
- Downstream layers (search, relationships, embeddings) operate on resolved entities.
- Entity resolution is auditable — every merge decision is recorded with confidence score and source.
- False positives can be detected and reversed through audit trail.

### Negative
- Import latency increases due to matching overhead.
- Resolution requires a matching strategy (fuzzy, semantic, exact) that must be tuned per domain.
- False positives may merge distinct entities. Mitigation: confidence thresholds and human review for low-confidence matches.

### Risks
- Resolution accuracy degrades with domain diversity. Different domains (climate policy vs. software tools vs. people) require different matching strategies. Mitigation: pluggable resolution strategies per entity type.
- Resolution at import time creates a coupling between import and entity store. Mitigation: the resolution layer is a separate service that the import layer calls, not embedded in the import logic.
