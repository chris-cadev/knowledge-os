# ADR-0007: Multi-Format Import via ImportAdapter Trait

**Status:** Accepted
**Date:** 2026-07-22
**Deciders:** Core maintainers

## Context

The initial import pipeline (PRD-0001) handles only Markdown files. Real knowledge is heterogeneous: PDFs sit alongside Markdown notes, remote URLs contain importable content, and users expect to import whatever format they have. The import layer must support multiple formats without coupling to any specific parser.

The system must also support importing from remote sources (URLs) in addition to local files.

## Decision

The import layer uses a trait-based adapter pattern. Each format is implemented as an `ImportAdapter` that conforms to a common interface. The CLI selects the adapter based on file extension or URL scheme. Adding a new format requires implementing a new adapter — no changes to the import pipeline or domain model.

```rust
#[async_trait]
pub trait ImportAdapter: Send + Sync {
    fn can_import(&self, path: &Path) -> bool;
    async fn import(&self, path: &Path) -> Result<ImportResult, ImportError>;
    fn supported_extensions(&self) -> &[&str];
}
```

The initial adapters are:
- **Markdown adapter:** Parses YAML frontmatter and Markdown body via `pulldown-cmark`.
- **PDF adapter:** Extracts text and metadata via `lopdf`. Produces `Content` (extracted text) and `BinaryContent` (file reference) components.

URL import is handled by a pre-processing step that downloads remote files to a temporary location, then delegates to the appropriate format adapter.

## Alternatives Considered

### Option 1: Format-specific import functions
- Pros: Simple, no abstraction overhead.
- Cons: Adding a new format requires modifying the import pipeline. Format logic leaks into orchestration code.
- Why not chosen: Violates open/closed principle. Couples format parsing to pipeline orchestration.

### Option 2: Single monolithic importer
- Pros: All format logic in one place.
- Cons: Growing complexity as formats are added. Testing requires exercising all formats. A bug in one format affects all others.
- Why not chosen: Creates a fragile, untestable monolith.

### Option 3: ImportAdapter trait (chosen)
- Pros: Each format is isolated. Adding a new format requires only a new adapter implementation. Adapters are independently testable. The CLI selects the adapter dynamically.
- Cons: Adds an abstraction layer. Each adapter must be implemented and maintained.
- Why chosen: Follows the adapter pattern established in ADR-0002 for storage. Consistent architectural approach.

## Consequences

### Positive
- Adding new formats (HTML, EPUB, plain text) requires only implementing a new adapter.
- Format-specific logic is isolated and independently testable.
- The import pipeline remains format-agnostic. Orchestration code never references specific formats.
- URL import is a pre-processing concern, not a format concern.

### Negative
- The adapter abstraction adds indirection. Debugging format-specific issues requires tracing through the trait.
- Each adapter must handle its own error cases (e.g., scanned PDFs, malformed Markdown).
- The adapter interface may need refinement as new formats reveal edge cases.

### Risks
- PDF text extraction quality varies across libraries. Mitigation: use battle-tested `lopdf`; fallback to `BinaryContent`-only for scanned PDFs.
- URL import introduces network dependencies. Mitigation: timeout handling, retry logic, and clear error messages for network failures.
