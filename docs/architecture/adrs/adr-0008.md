# ADR-0008: Fuzzy Entity Resolution with Confidence Scoring

**Status:** Accepted
**Date:** 2026-07-22
**Deciders:** Core maintainers

## Context

ADR-0006 established that entity resolution is the critical quality layer and must happen before storage. The initial implementation uses exact match resolution (title + entity type equality). However, real knowledge bases contain duplicates that differ by casing, punctuation, abbreviations, or minor wording changes — "Attention Is All You Need" vs. "attention is all you need" vs. "Attention Is All You Need (2017)".

Without fuzzy resolution, the system catches only exact duplicates. The 2026 landscape evidence confirms that entity resolution quality is where knowledge graphs succeed or fail, and fuzzy matching is essential for catching the majority of real-world duplicates.

## Decision

Entity resolution uses a composable strategy pattern with confidence scoring. Each strategy produces a confidence score (0.0–1.0) for each candidate. Strategies are tried in order of increasing cost and decreasing precision. The resolver stops when a strategy produces a match above the configurable threshold for the entity type.

Resolution strategies:

| Strategy         | Method                           | Confidence Source          | Use Case                       |
| ---------------- | -------------------------------- | -------------------------- | ------------------------------ |
| Exact match      | Title + entity type equality     | Fixed at 1.0               | Fast, high confidence          |
| Normalized match | Lowercase + whitespace normalize | Fixed at 0.95              | Catches casing differences     |
| Fuzzy match      | Levenshtein / Jaro-Winkler       | Distance-based (0.0–1.0)   | Catches typos, abbreviations   |
| Content match    | Body text similarity             | Similarity-based (0.0–1.0) | Same content, different titles |

Each entity type has a configurable merge threshold. Matches above the threshold are merged automatically. Matches below the threshold are flagged for review. The default threshold is 0.85.

Every merge decision is logged with:
- Source entity ID and title
- Target (canonical) entity ID and title
- Strategy that produced the match
- Confidence score
- Timestamp
- Reason (automatic vs. manual)

Merged entities can be undone by replaying the audit trail — the merged entity is restored and the canonical entity reverts to its pre-merge state.

## Alternatives Considered

### Option 1: Exact match only
- Pros: Simple, zero false positives.
- Cons: Misses duplicates with minor variations. The majority of real-world duplicates are fuzzy, not exact.
- Why not chosen: Insufficient coverage. Catches < 30% of real duplicates.

### Option 2: Embedding-based semantic resolution
- Pros: Catches semantically similar entities regardless of wording.
- Cons: Requires vector storage, model inference, and threshold tuning. High complexity. Defers to PRD-0003.
- Why not chosen: Too complex for initial implementation. Fuzzy string matching provides sufficient coverage.

### Option3: Composable fuzzy strategies (chosen)
- Pros: Each strategy is independently testable. Strategies are composable — exact first, then fuzzy. Confidence scoring enables configurable thresholds. Audit trail enables undo.
- Cons: Requires tuning thresholds per entity type. Fuzzy matching has O(n) cost per candidate set.
- Why chosen: Balances accuracy, performance, and auditability. Follows the trait-based pattern established in ADR-0002 and ADR-0007.

## Consequences

### Positive
- Duplicates with minor variations are caught automatically.
- Confidence scoring enables fine-grained control over merge behavior.
- Different entity types (Person, Concept, Article) can have different thresholds.
- Every merge decision is auditable and undoable.
- The strategy pattern is extensible — new strategies (e.g., phonetic matching) can be added without changing the resolver.

### Negative
- Fuzzy matching adds computational cost to the import pipeline.
- Threshold tuning requires domain-specific calibration.
- False positives may merge distinct entities. Mitigation: configurable thresholds and human review for low-confidence matches.

### Risks
- Resolution performance degrades with large candidate sets. Mitigation: index-based lookup, bounded candidate sets (top-K by title similarity).
- Threshold tuning is manual and error-prone. Mitigation: sensible defaults per entity type, documented tuning guidelines.
