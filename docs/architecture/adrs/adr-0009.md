# ADR-0009: Extended Cross-Reference Patterns

**Status:** Accepted
**Date:** 2026-07-22
**Deciders:** Core maintainers

## Context

The initial import pipeline (PRD-0001) extracts standard Markdown links (`[text](path)`) as cross-references. Real knowledge documents use richer reference patterns: wikilinks (`[[name]]`) for internal references, URLs for external resources, `@mentions` for people or organizations, and section anchors (`file.md#section`) for precise linking.

Without extended cross-reference extraction, the relationship engine captures only a fraction of the semantic connections between entities. The knowledge graph is sparse where it should be dense.

## Decision

The cross-reference extraction system supports multiple reference patterns. Each pattern produces a `references` relationship between entities, using the existing relationship type from the domain model.

Supported patterns:

| Pattern        | Syntax                | Example                         | Produces                           |
| -------------- | --------------------- | ------------------------------- | ---------------------------------- |
| File link      | `[text](path)`        | `[paper](papers/ai.md)`         | `references` to target entity      |
| URL link       | `[text](https://...)` | `[Google](https://google.com)`  | `references` with URL metadata     |
| Wikilink       | `[[name]]`            | `[[Attention Is All You Need]]` | `references` to named entity       |
| @mention       | `@name`               | `@geoffrey_hinton`              | `references` to Person entity      |
| Section anchor | `path#section`        | `paper.md#methodology`          | `references` with section metadata |

The `CrossReference` type captures all variants:

```rust
pub enum CrossReference {
    FileRef { target_path: PathBuf, link_text: String },
    UrlRef { url: String, link_text: String },
    WikilinkRef { target_name: String, link_text: String },
    MentionRef { target_name: String },
    SectionRef { target_path: PathBuf, section: String, link_text: String },
}
```

Wikilinks require a pre-processing pass before pulldown-cmark, since they are not standard Markdown syntax. The extractor uses regex to find `[[...]]` patterns and converts them to entity name lookups.

All cross-references are resolved to existing entities during the relationship engine phase (Layer 5). Unresolved references are logged as warnings but do not block import.

## Alternatives Considered

### Option 1: Standard Markdown links only
- Pros: Simple, relies on existing parser.
- Cons: Misses wikilinks, @mentions, and URLs that are not standard Markdown links. The relationship graph is sparse.
- Why not chosen: Insufficient coverage. Real documents use diverse reference patterns.

### Option 2: AI-assisted reference extraction
- Pros: Catches implicit references that regex cannot.
- Cons: Requires LLM inference, adds latency, non-deterministic. Deferred to Year 2.
- Why not chosen: Too complex for initial implementation. Regex-based extraction provides sufficient coverage.

### Option3: Multi-pattern regex extraction (chosen)
- Pros: Deterministic, fast, extensible. Each pattern is independently testable. All patterns produce the same `references` relationship type.
- Cons: Regex patterns may miss edge cases. Unresolved references require logging.
- Why chosen: Balances coverage, performance, and determinism. Follows the parsing layer pattern from ADR-0005.

## Consequences

### Positive
- The relationship graph captures more semantic connections between entities.
- Wikilinks enable wiki-style knowledge navigation.
- @mentions enable people-centric graph traversal.
- URLs enable external reference tracking.
- Section anchors enable precise intra-document referencing.

### Negative
- Wikilink syntax varies across tools (Obsidian, Roam, etc.). Mitigation: support common patterns, document limitations.
- Unresolved references add noise to the relationship graph. Mitigation: log warnings, do not block import.
- Regex-based extraction may miss complex or nested references. Mitigation: AI-assisted extraction deferred to Year 2.

### Risks
- Wikilink syntax inconsistency across tools. Mitigation: configurable regex patterns per wikilink dialect.
- Performance impact of multiple regex passes. Mitigation: single-pass extraction with combined patterns.
