# ADR-0010: Entity Type Inference from Frontmatter

**Status:** Accepted
**Date:** 2026-07-22
**Deciders:** Core maintainers

## Context

The initial import pipeline (PRD-0001) assigns entity type based on content analysis or defaults to `Article` for Markdown files. Users have diverse knowledge: papers, books, tools, people, concepts. Forcing everything into `Article` loses semantic meaning and degrades the quality of type-based queries.

The domain model defines 20 entity types (Concept, Person, Organization, Project, Book, Paper, Video, Article, Tool, Technology, Question, Idea, Event, Skill, Location, Dataset, Collection, Workspace, Decision, Note). Users should be able to specify which type applies to their content.

## Decision

The importer parses a `type` field from YAML frontmatter. The value is matched against `EntityType` variants (case-insensitive). If the `type` field is absent or invalid, the default type (`Article`) is used.

```yaml
---
title: "Attention Is All You Need"
type: paper
tags: [transformer, attention]
---
```

This is a simple extension of the existing frontmatter extraction pattern. The importer already extracts `title`, `tags`, `author`, `language` — adding `type` is consistent with the established pattern.

The `type` field is optional. The default type is `Article` for Markdown and `Article` for PDF (unless overridden by PDF metadata).

## Alternatives Considered

### Option 1: Content-based type inference
- Pros: Automatic, no user effort.
- Cons: Requires AI or heuristic analysis. Non-deterministic. High false positive rate.
- Why not chosen: Too complex and unreliable for initial implementation. Deferred to Year 2.

### Option 2: File extension mapping
- Pros: Simple, automatic for format-specific types.
- Cons: Limited to format-level types (e.g., `.pdf` → Paper). Does not capture content-level types.
- Why not chosen: Insufficient granularity. A PDF could be a Paper, Book, or Article.

### Option 3: Frontmatter `type` field (chosen)
- Pros: Explicit, deterministic, user-controlled. Consistent with existing frontmatter pattern. Zero false positives.
- Cons: Requires user to set the field. Default type may not be correct.
- Why chosen: Simple, reliable, and consistent with the existing architecture. Users who care about type can set it; those who don't get a sensible default.

## Consequences

### Positive
- Users can explicitly type their entities at import time.
- Type-based queries return semantically correct results.
- The frontmatter pattern is consistent with existing metadata extraction.
- Zero false positives — the type is explicitly declared, not inferred.

### Negative
- Users must set the `type` field manually. This is an extra step.
- Invalid `type` values are silently ignored (default to `Article`). Mitigation: log a warning for invalid types.

### Risks
- Users may not know which type to use. Mitigation: document the available types and provide examples in the CLI help.
