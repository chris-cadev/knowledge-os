# ADR-0011: BinaryContent Component

**Status:** Accepted
**Date:** 2026-07-22
**Deciders:** Core maintainers

## Context

The initial import pipeline (PRD-0001) handles only Markdown, which is pure text. PDF import (PRD-0002) introduces binary content — the original PDF file must be retained alongside extracted text. Without a component to reference the binary file, the system either loses the original or stores it in the canonical model, violating the separation between text and binary data.

The `Content` component holds extracted text (Markdown body or PDF text extraction). But the original binary file — the PDF itself — needs a reference in the entity model. This reference enables future operations: re-extraction with better tools, binary rendering, or export.

## Decision

The `BinaryContent` component type is added to the core component types in the domain model. It stores a reference to binary data in object storage, not the binary data itself.

```rust
pub struct BinaryContent {
    pub reference: String,  // Path or key in object storage
    pub mime_type: String,  // MIME type (e.g., "application/pdf")
    pub size: u64,          // Size in bytes
}
```

The component follows the same rules as all core component types:
- An entity may have at most one `BinaryContent` component.
- The component is versioned with its entity.
- The component is data only — behavior lives in systems.

PDF import produces entities with both `Content` (extracted text) and `BinaryContent` (reference to original PDF). Markdown import produces entities with `Content` only (no binary backing).

## Alternatives Considered

### Option 1: Store binary data in the canonical model
- Pros: Simple, all data in one place.
- Cons: Canonical model bloat. Binary data cannot be versioned meaningfully. Storage engine must handle large blobs.
- Why not chosen: Violates the separation between canonical text data and binary blobs.

### Option 2: Store binary data in a separate file system without a component
- Pros: No canonical model changes.
- Cons: Binary data is disconnected from the entity model. No way to reference it from the entity. No lifecycle management.
- Why not chosen: Loses the relationship between entity and binary data.

### Option 3: BinaryContent component with object storage reference (chosen)
- Pros: The entity model references binary data without storing it. Binary data is lifecycle-managed through the component. The component follows existing patterns.
- Cons: Requires an object storage backend. Binary data is not in the canonical model.
- Why chosen: Clean separation between text and binary. The component is a reference, not a container.

## Consequences

### Positive
- The entity model can reference binary files without storing them.
- Binary data is lifecycle-managed through the component system.
- Multiple entities can reference the same binary file (e.g., a PDF and its extracted text).
- The component follows existing patterns and invariants.

### Negative
- Requires an object storage backend (file system initially, S3 later).
- Binary data is not versioned with the entity — only the reference is versioned.
- Re-extraction from binary requires accessing the storage backend.

### Risks
- Object storage backend may not be available in all deployment contexts. Mitigation: use local file system as the initial backend; abstract through a storage trait.
