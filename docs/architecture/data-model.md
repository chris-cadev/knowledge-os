# Data Model

> Canonical data cannot be recreated. Derived data may be regenerated at any time.

---

## The Distinction

The most important architectural decision in Knowledge OS is the separation of canonical data from derived data. This distinction determines what is durable, what is disposable, what is the source of truth, and what is a projection.

---

## Canonical Data

Canonical data represents user knowledge. It cannot be recreated from other sources. It requires durability, versioning, and auditing.

**Characteristics:**

- Cannot be recreated
- Represents user knowledge
- Requires durability
- Versioned
- Auditable
- The source of truth

**Examples:**

- Entities (people, papers, concepts, tools)
- Components (titles, descriptions, content, metadata)
- Relationships (created_by, references, depends_on)
- Artifacts (imported documents, images, code)
- Provenance (source attribution, import history)

---

## Derived Data

Derived data is generated by computation from canonical data. It optimizes specific access patterns. It is never authoritative. It may be discarded and reconstructed at any time.

**Characteristics:**

- May be recreated
- Optimizes performance, search, AI, navigation
- Never becomes authoritative
- Disposable
- Reproducible

**Examples:**

- Search indexes (inverted indexes for full-text retrieval)
- Embeddings (vector representations for semantic similarity)
- Similarity graphs (computed proximity between entities)
- Recommendations (AI-generated suggestions)
- Learning paths (ordered sequences for progression)
- Knowledge summaries (condensed representations)
- AI context (retrieval-augmented generation payloads)
- Caches (temporary performance optimization)

---

## Why This Matters

Without this distinction, systems suffer from a fundamental problem: derived data becomes authoritative. A search index becomes the only way to find information. An embedding becomes the only representation of meaning. A cache becomes the only source of truth.

When derived data becomes authoritative, it cannot be rebuilt. Changing the embedding model requires rebuilding the entire index. Changing the search engine requires re-indexing everything. Changing the graph structure requires recomputing all relationships.

Knowledge OS avoids this by making the distinction explicit. Canonical data is never derived. Derived data is never canonical.

---

## The Compiler Analogy

This distinction mirrors the relationship between source code and compiled artifacts in compiler architecture:

| Compiler Concept            | Knowledge OS Equivalent                       |
| --------------------------- | --------------------------------------------- |
| Source code                 | Canonical entities, relationships, components |
| Intermediate representation | Normalized knowledge model                    |
| Compiled artifacts          | Search indexes, embeddings, graphs            |
| Executable                  | Rendered views                                |
| Object files                | Cached projections                            |

Source code is the source of truth. Object files are derived. If you lose the object files, you recompile. If you lose the search index, you rebuild it. The canonical model is the source code of knowledge.

---

## Regeneration

Every derived artifact supports regeneration:

1. **Search indexes** are rebuilt from canonical content and metadata.
2. **Embeddings** are recomputed from canonical text fields.
3. **Similarity graphs** are recalculated from embeddings.
4. **Recommendations** are regenerated from the knowledge graph.
5. **Learning paths** are recomputed from relationship graphs.
6. **Caches** are populated on demand from canonical data.

Regeneration may be triggered by:

- Canonical data changes
- Configuration changes (new embedding model, new search engine)
- Manual rebuild commands
- System recovery after failure

## Storage Mapping

Canonical and derived data map to different storage strategies:

| Data Category     | Storage Type        | Persistence | Rebuildable |
| ----------------- | ------------------- | ----------- | ----------- |
| Entities          | Relational          | Durable     | No          |
| Components        | Relational + Object | Durable     | No          |
| Relationships     | Graph + Relational  | Durable     | Partially   |
| Artifacts         | Object storage      | Durable     | No          |
| Search indexes    | Search engine       | Volatile    | Yes         |
| Embeddings        | Vector store        | Volatile    | Yes         |
| Similarity graphs | Graph store         | Volatile    | Yes         |
| Caches            | Cache store         | Volatile    | Yes         |

---

## Implications for Engineering

When building a new feature, ask:

1. Is the data canonical or derived?
2. If canonical, how is it versioned and audited?
3. If derived, can it be regenerated?
4. If derived, what triggers regeneration?
5. If derived, does it contain any information that cannot be recovered from canonical sources?

If a derived artifact contains information that cannot be recovered, it is misclassified. Reclassify it as canonical.
