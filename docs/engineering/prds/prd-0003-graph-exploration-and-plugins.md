# PRD-0003: Graph Exploration, Views, and Plugin System

**Status:** Draft
**Date:** 2026-07-22
**Author:** Core maintainers
**Priority:** P0 — Experience Layer
**Depends on:** PRD-0001, PRD-0002

---

## Purpose

This PRD transforms the knowledge graph from a backend into an explorable experience. It adds multi-hop relationship traversal, visual projections (tree, graph, table), and the plugin system that enables extensibility. This is the knowledge graph experience — it makes the system usable and growable.

---

## Problem Statement

PRD-0001 builds the graph. PRD-0002 makes it trustworthy. But a knowledge graph you can't explore is just a database. Users need to navigate relationships, view entities in context, and extend the system with custom importers and renderers.

The current CLI provides basic CRUD and search. It does not provide traversal, visual projections, or extensibility. Without these, the system demonstrates the thesis but does not deliver the experience.

- AI-assisted knowledge construction (Year 2)

## Engineering Questions

Every feature proposed for the system must answer these 10 questions per [Engineering Principles](../../philosophy/engineering-principles.md):

### 1. Which canonical entities are introduced?

One new entity type is introduced:

| Type         | Description                 | Example                      |
| ------------ | --------------------------- | ---------------------------- |
| `Collection` | A curated group of entities | "Papers to read," "AI tools" |

Collections are used by view projections to group entities for navigation.

### 2. Which relationships are introduced?

No new relationship types are introduced. The system uses the core relationship types from `docs/architecture/domain-model.md`. The `contains` and `belongs_to` relationships are used for collection membership.

### 3. Which components are introduced?

One new component type is introduced:

| Component   | Source                                  |
| ----------- | --------------------------------------- |
| `Embedding` | AI-generated vector for semantic search |

Embeddings are generated by configurable AI providers (plugins). The component stores the vector, model name, and generation timestamp.

### 4. Which events are emitted?

| Event               | Trigger                      | Consumers          |
| ------------------- | ---------------------------- | ------------------ |
| `ViewUpdated`       | View projection data changed | CLI, future UI     |
| All PRD-0001 events | Existing pipeline events     | Existing consumers |

### 5. Which derived representations are generated?

| Derived Artifact         | Source                        | Regeneration              |
| ------------------------ | ----------------------------- | ------------------------- |
| Tree view projection     | Entity type + collection data | Rebuild from canonical    |
| Graph view projection    | Entity + relationship data    | Rebuild from canonical    |
| Table view projection    | Entity + component data       | Rebuild from canonical    |
| Timeline view projection | Timeline component            | Rebuild from canonical    |
| Embedding index          | Entity Content components     | Regenerate from canonical |

All derived data remains disposable and rebuildable from canonical data.

### 6. Which layer owns the feature?

| Feature          | Layer                                      |
| ---------------- | ------------------------------------------ |
| N-hop traversal  | Layer 5 — Relationship Engine              |
| View projections | Layer 6 — Derivation + Layer 7 — Rendering |
| Plugin system    | Layer 0 — Infrastructure (cross-cutting)   |
| Semantic search  | Layer 6 — Derivation                       |

### 7. Can every derived artifact be regenerated?

Yes. All view projections are rebuilt from canonical entity and relationship data. The embedding index is regenerated from canonical Content components.

### 8. Does the feature violate storage independence?

No. Traversal uses the existing adapter interface. View projections are derived data. The plugin system is infrastructure that operates above the storage layer. Vector storage uses a pluggable adapter.

### 9. Does the feature introduce implementation leakage?

No. View projections are trait-based and swappable. The plugin system uses a manifest and capability negotiation, not runtime reflection. Traversal is bounded by the adapter interface.

### 10. Does the feature preserve the canonical model?

Yes. The canonical model is the source of truth. View projections are disposable. Plugins extend the system without modifying the core. Traversal reads canonical data without modifying it.

## Pipeline Spine Analysis

The system spine remains: **Import → Extract → Resolve → Store → Connect → Search**

PRD-0003 adds downstream capabilities that consume the canonical model.

### Pipeline Extension

```
Canonical Model (PRD-0001 + PRD-0002)
      |
  Relationship Engine  N-hop traversal (bounded recursive CTE)
      |
  Derivation Layer     View projections (tree, graph, table, timeline)
                       Embedding generation (AI provider plugin)
      |
  Rendering Layer      CLI output formatting
      |
  Plugin System        Importer plugins, renderer plugins, AI provider plugins
```

### Why Traversal Is Bounded

N-hop traversal is bounded by design:

- Default depth: 3 hops
- Maximum depth: configurable, capped at 10
- Performance target: < 100ms for 2 hops at 100K entities (NF1.1)
- Recursive CTE in SQLite provides efficient bounded traversal
- Unbounded traversal is explicitly avoided — it is the path to performance collapse

## User Stories

### US1: Traverse Entity Relationships

**As a** knowledge worker,
**I want to** explore connections between entities,
**So that** I can understand how concepts relate to each other.

**Acceptance criteria:**
1. User runs `kos traverse <entity-id> --depth 2`.
2. System returns all entities within 2 hops.
3. Results include entities and relationships.
4. Cycle detection prevents infinite loops.
5. User can filter by relationship type (`--type references`).
6. User can filter by entity type (`--entity-type concept`).

### US2: View Entities as a Tree

**As a** knowledge worker,
**I want to** browse my entities hierarchically,
**So that** I can navigate my knowledge by category.

**Acceptance criteria:**
1. User runs `kos view tree`.
2. System displays entities grouped by type.
3. Collections appear as branches containing their members.
4. User can expand/collapse branches.
5. User can filter by type (`--type paper`).

### US3: View Entities as a Graph

**As a** knowledge worker,
**I want to** see my entities as a graph,
**So that** I can visualize connections and clusters.

**Acceptance criteria:**
1. User runs `kos view graph`.
2. System displays entities as nodes, relationships as edges.
3. Nodes are labeled with entity type and title.
4. Edges are labeled with relationship type.
5. User can filter by type and tags.
6. User can view a subgraph from a specific entity (`kos view graph --from <entity-id> --depth 2`).

### US4: View Entities as a Table

**As a** knowledge worker,
**I want to** see my entities in a sortable table,
**So that** I can compare and filter entities efficiently.

**Acceptance criteria:**
1. User runs `kos view table`.
2. System displays entities with columns: ID, Type, Title, Tags, Created, Updated.
3. User can sort by any column.
4. User can filter by type, tags, and text.
5. User can combine with search query.

### US5: Use a Plugin Importer

**As a** plugin developer,
**I want to** create a custom importer plugin,
**So that** I can add support for new file formats without modifying the core.

**Acceptance criteria:**
1. Developer creates a TOML manifest declaring `importer` capability.
2. Developer implements `ImportAdapter` trait.
3. System discovers and loads the plugin at startup.
4. Plugin importer is available via CLI (`kos import <file>`).
5. Plugin failure does not crash the core system.

### US6: Semantic Search

**As a** knowledge worker,
**I want to** search by meaning, not just keywords,
**So that** I can find conceptually related entities.

**Acceptance criteria:**
1. User runs `kos search "machine learning" --semantic`.
2. System returns entities semantically similar to the query.
3. Results include entities that are topically related but don't contain the exact keywords.
4. User can combine semantic and keyword search (`--hybrid`).
5. Embedding provider is configurable.

### Plugin System

The plugin system is infrastructure that enables extensibility without modifying the core.

#### Plugin Manifest

Every plugin declares a TOML manifest:

```toml
[plugin]
name = "markdown-importer"
version = "0.1.0"
description = "Import Markdown files as knowledge entities"
author = "Knowledge OS"

[plugin.capabilities]
importers = ["markdown"]

[plugin.dependencies]
knowledge-os = ">=0.1.0"
```

#### Plugin Loading

```
Startup
  |
  Scan plugin directories (configurable)
  |
  Parse manifests
  |
  Validate capabilities and dependencies
  |
  Load dynamic libraries (libloading)
  |
  Register with capability registry
  |
  Activate plugins
```

#### Capability Registry

The capability registry maps capabilities to plugins:

```rust
pub struct CapabilityRegistry {
    importers: HashMap<String, Box<dyn ImportAdapter>>,
    renderers: HashMap<String, Box<dyn RenderAdapter>>,
    ai_providers: HashMap<String, Box<dyn AiAdapter>>,
}
```

When the system needs a capability, it queries the registry:

```
System needs: import("markdown")
Registry: markdown-importer v0.1.0 (priority: 10)
Selected: markdown-importer
```

#### Plugin Sandboxing

Plugins run in the same process but are isolated through:

- Trait-based interfaces (no shared mutable state)
- Error boundaries (plugin errors are caught, not propagated)
- Resource limits (configurable per plugin)
- Timeout limits (configurable per plugin)

### Graph Traversal

N-hop traversal uses recursive CTEs in SQLite:

```sql
WITH RECURSIVE traverse(id, depth) AS (
    SELECT id, 0 FROM entities WHERE id = ?1
    UNION
    SELECT r.target_id, t.depth + 1
    FROM relationships r
    JOIN traverse t ON r.source_id = t.id
    WHERE t.depth < ?2
      AND r.is_active = 1
)
SELECT DISTINCT e.*, t.depth
FROM traverse t
JOIN entities e ON e.id = t.id
WHERE e.is_active = 1
ORDER BY t.depth;
```

The traversal is bounded by the depth parameter. Cycle detection is handled by `SELECT DISTINCT`.

### View Projections

Each view is a `ViewAdapter` trait implementation:

```rust
#[async_trait]
pub trait ViewAdapter: Send + Sync {
    fn name(&self) -> &str;
    async fn render(&self, filter: &ViewFilter) -> Result<ViewOutput>;
    async fn synchronize(&self, event: &Event) -> Result<()>;
}
```

Built-in views:

| View     | Grouping               | Output Format      |
| -------- | ---------------------- | ------------------ |
| Tree     | Entity type → entities | Indented text      |
| Graph    | Nodes + edges          | Adjacency list     |
| Table    | Flat rows              | Tabular text       |
| Timeline | Temporal order         | Chronological list |

Views are updated event-driven. When canonical data changes, the relevant view adapters are notified via the event system.

### Semantic Search

Embedding generation is delegated to AI provider plugins:

```rust
#[async_trait]
pub trait AiAdapter: Send + Sync {
    async fn embed(&self, content: &str) -> Result<Vec<f64>>;
    fn model_name(&self) -> &str;
}
```

The embedding pipeline:

1. Content component is extracted from entity
2. AI provider plugin generates embedding vector
3. Embedding component is attached to entity
4. Vector is indexed for similarity search
5. Search combines BM25 (keyword) and cosine similarity (semantic)

Vector storage uses a pluggable adapter. Initial implementation uses in-memory brute-force search. Production implementation uses Qdrant or similar.

### Storage Changes

| Table                | Purpose                                      |
| -------------------- | -------------------------------------------- |
| `collections`        | Collection entities (new entity type)        |
| `collection_members` | Many-to-many: collection ↔ entity            |
| `embeddings`         | Embedding vectors (entity_id, vector, model) |

Schema additions:

```sql
CREATE TABLE collections (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    description TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

CREATE TABLE collection_members (
    collection_id TEXT NOT NULL,
    entity_id TEXT NOT NULL,
    added_at TEXT NOT NULL,
    PRIMARY KEY (collection_id, entity_id)
);

CREATE TABLE embeddings (
    id TEXT PRIMARY KEY,
    entity_id TEXT NOT NULL,
    vector BLOB NOT NULL,
    model TEXT NOT NULL,
    created_at TEXT NOT NULL
);
```

## Risks and Mitigations

| Risk                                | Impact | Likelihood | Mitigation                                               |
| ----------------------------------- | ------ | ---------- | -------------------------------------------------------- |
| Recursive CTE performance at scale  | High   | Medium     | Bounded depth; index on relationship source_id/target_id |
| Plugin dynamic loading complexity   | Medium | Medium     | Start with in-process plugins; dynamic loading deferred  |
| Embedding quality variance          | Medium | Medium     | Configurable provider; fallback to keyword search        |
| View synchronization staleness      | Low    | Low        | Event-driven updates; eventual consistency acceptable    |
| Plugin security (malicious plugins) | High   | Low        | Sandboxing; resource limits; community review            |
