# Glossary

> Every project term, defined once. All documents reference this glossary.

---

## A

**Adapter**
A module that isolates a specific storage engine, AI provider, or external service from the domain model. Adapters implement a common interface. The domain model never depends on a specific implementation.

**Agent**
An AI-powered component that performs autonomous tasks within the knowledge system. Agents suggest actions for human review. Agents are scoped, auditable, and replaceable. See also: Automation Agent.

**Artifact**
A piece of content imported into the system. Artifacts are canonical data. They include documents, images, audio, video, code, and any other imported material.

**Automation**
The process of performing tasks within the knowledge system without direct human intervention for each operation. Automation is opt-in, auditable, and bounded by explicit rules. See also: Agent.

---

## C

**Canonical Data**
Data that represents the authoritative source of truth. Canonical data cannot be recreated from other sources. It requires durability, versioning, and auditing. Examples: entities, relationships, components, artifacts.

**Canonical Entity Model**
The central representation of all knowledge in the system. The canonical entity model is storage-independent and technology-independent. It is the single source of truth.

**Capability**
A declaration of what a plugin provides. Capabilities are typed: importer, exporter, renderer, storage, search, vector, graph, cache, ai, automation, view. The system queries capabilities to determine which plugins handle which operations.

**Collection**
A group of entities organized for a specific purpose. Collections are themselves entities. They may be manually curated or dynamically generated.

**Component**
A data structure that represents one aspect of an entity. Components are composable and reusable. Examples: Title, Description, Content, Author, Tags, Timeline, Embedding.

**Compiler Perspective**
The architectural model where knowledge processing follows the same stages as source code compilation: import, parse, normalize, derive, render.

**Context**
The information assembled for a specific interaction, query, or AI operation. Context is derived from canonical data and may include entity summaries, relationship graphs, and retrieved documents.

---

## D

**Derived Data**
Data generated by computation from canonical data. Derived data optimizes specific access patterns but is never authoritative. It may be discarded and rebuilt at any time. Examples: search indexes, embeddings, caches.

**Domain Model**
The conceptual representation of the knowledge system's problem space. The domain model defines entities, relationships, components, and the rules that govern them.

---

## E

**Entity**
A first-class knowledge object in the system. Entities represent real-world concepts: people, organizations, papers, tools, ideas, events. Every entity is composed of components. Entities are canonical data.

**Entity Component Model**
An architectural pattern where entities are assembled from composable components rather than defined through inheritance hierarchies. Adapted from game engine ECS architecture.

**Event**
A record of a meaningful change to canonical data. Events trigger asynchronous processing through the derivation pipeline. Events are immutable and ordered.

---

## G

**Graph Storage**
A storage engine optimized for relationship traversal. Graph storage represents one projection of canonical relationships. It may be rebuilt from relational canonical data.

---

## I

**Importer**
A plugin that receives information from an external system and transforms it into an internal representation. Importers are format-specific. New formats are added as plugins.

---

## K

**Knowledge Engine**
The core system that processes, stores, and serves knowledge. Knowledge OS is a knowledge engine, not a database. The application owns the knowledge model.

**Knowledge Graph**
The emergent structure that arises from all entities and relationships in the system. The knowledge graph is heterogeneous, attributed, versioned, evolving, and queryable. It is not a separate storage engine.

**Knowledge Model**
The canonical representation of all knowledge entities, their components, and their relationships. The knowledge model is the single source of truth.

---

## M

**Metadata**
Data about data. In Knowledge OS, metadata includes entity provenance (source, import time, importer), relationship attributes (confidence, source, timestamp), and component metadata (model version, generation time). Metadata is canonical data.

---

## P

**Pipeline**
The sequence of layers through which information flows: import, parsing, normalization, knowledge model, relationship engine, derivation, presentation. The pipeline is deterministic and composable.

**Plugin**
A module that extends the system without modifying the core. Plugins implement adapters for storage engines, importers, exporters, views, AI providers, and automation agents.

**Presentation Layer**
The layer that renders projections of knowledge for human and machine interfaces. Views are projections. No view owns data.

**Projection**
A view of canonical data optimized for a specific access pattern. Projections are derived data. They may be discarded and rebuilt. Examples: search indexes, graph views, rendered pages.

---

## R

**Relationship**
A typed, directed, attributed edge connecting two entities. Relationships are first-class citizens with their own metadata, versioning, and queryability.

**Renderer**
A plugin that transforms canonical data into a specific output format. Examples: HTML renderer, JSON API renderer, MCP server renderer.

**Resource**
An external system, service, or data source that Knowledge OS interacts with. Resources include storage engines, AI providers, external APIs, and imported data sources. Resources are accessed through adapters.

---

## S

**Storage Engine**
A technology that persists data. Storage engines are interchangeable adapters. They optimize access patterns. They do not define meaning.

**Source of Truth**
The authoritative data from which all other representations are derived. In Knowledge OS, the canonical knowledge model is the source of truth.

**Synchronization**
The process of keeping derived data consistent with canonical data. Synchronization is event-driven and eventually consistent. Derived data may briefly lag behind canonical data.

---

## V

**Vector Storage**
A storage engine optimized for semantic retrieval. Vector storage stores embeddings and supports approximate nearest neighbor search.

**View**
A projection of canonical data rendered for a specific interface. Views never own data. Views remain synchronized because they render canonical data.

**Version**
A snapshot of an entity or relationship at a specific point in time. Versions enable audit trails, rollback, and temporal queries.

---

## W

**Workspace**
A bounded context within the knowledge system. Workspaces scope access, collaboration, and organization. Workspaces are themselves entities.
